# ADR-0009 — phase-07 인증 아키텍처 (Supabase Auth · 이메일+Google · 앱-레벨 프로필)

**상태** Accepted
**날짜** 2026-05-18
**관련** `docs/adr/0001-tech-stack.md`(§2.2 Supabase Auth 결정), `docs/adr/0003-supabase-new-api-keys.md`(publishable/secret 키 표준), `supabase/migrations/001_initial_schema.sql`(profiles 테이블·RLS), `docs/intent/auth-flow.md`, `tasks/phase-07-auth.json`, `PLAN.md` 9절(Week 3~4 인증), `claude.md` 2절 Hard Rule 6·8·10

---

## 1. 배경

ADR-0001 §2.2는 인증 수단으로 "Supabase Auth — 이메일·구글·카카오 OAuth"를 정했다. 그러나 ADR-0001은 스택 선정 수준의 결정이고, 실제 phase-07 구현에는 더 구체적인 결정 네 가지가 필요하다.

1. 베타에서 어떤 로그인 수단까지 구현할 것인가
2. 로그인한 사용자의 `profiles` 행을 누가 만들 것인가
3. 세션을 어떻게 유지·갱신할 것인가
4. 비로그인 사용자의 보호 페이지 접근을 어디서 막을 것인가

특히 (2)는 claude.md Hard Rule 8("DB 스키마 변경 시 신규 ADR 작성 선행")과 직결된다 — 흔히 쓰는 `handle_new_user` 트리거 방식은 스키마 변경을 수반하기 때문이다. 본 ADR은 네 결정을 기록하고 각 대안을 비교한다.

---

## 2. 결정

1. **인증 공급자 범위** — phase-07은 **이메일(비밀번호) + Google OAuth**만 구현한다. **카카오는 phase-13 이후로 연기**한다.
2. **프로필 생성 방식** — `profiles` 행은 **앱 코드의 idempotent upsert**(`ensureProfile` 헬퍼)로 만든다. **`handle_new_user` 등 DB 트리거는 채택하지 않는다.** → DB 스키마 변경 없음.
3. **세션 관리** — **Supabase SSR(`@supabase/ssr`)의 쿠키 기반 세션**을 사용하고, `middleware.ts`가 매 요청에서 세션을 갱신한다.
4. **보호 라우트** — **`middleware.ts` 중앙 가드**로 보호/공개 라우트를 한 곳에서 판정한다. 데이터 차원의 보호는 RLS가 최종 방어선이다.

---

## 3. 근거

### 3.1 인증 공급자 범위 — 이메일 + Google (카카오 연기)

| 후보 | 장점 | 탈락/연기 사유 |
|---|---|---|
| **이메일 + Google (선택)** | 소셜 로그인 흐름(OAuth 콜백·세션)을 Google로 완전 검증. 이메일 경로로 비밀번호 흐름도 검증. 베타 테스터 대부분 Google 계정 보유 | — |
| 이메일 + Google + 카카오 (ADR-0001 원안) | 한국 사용자 친화 | 카카오는 비즈니스 앱 등록·검수·도메인 확인 등 별도 절차 필요(ADR-0001 §5.2가 이미 경고). 베타 일정 압박. 베타는 기능 검증이 목적이라 필수 아님 |
| 이메일만 | 외부 설정 0 | 소셜 로그인 미검증. 학부모 가입 이탈률↑ |
| Google만 | 가장 단순 | 이메일 없는 사용자 배제, 비밀번호 흐름(재설정 등 후속 작업의 토대) 미검증 |

카카오 연기는 **기술 부채가 아니다.** 공급자 추가는 Supabase 대시보드 설정 + 로그인 화면 버튼 1개 수준이라, phase-13 이후 독립 작업으로 안전하게 붙일 수 있다. 베타 검증 가치 대비 설정 비용이 가장 낮은 조합이 이메일 + Google이다.

### 3.2 프로필 생성 방식 — 앱-레벨 upsert vs DB 트리거

| 후보 | 장점 | 단점 |
|---|---|---|
| **앱-레벨 idempotent upsert (선택)** | ① DB 스키마 변경 0 → Hard Rule 8의 마이그레이션·승인 절차 불필요 ② `001` 스키마에 이미 있는 RLS 정책 `users can insert own profile`이 정확히 이 용도 — 스키마 설계 의도와 일치 ③ "프로필이 언제 만들어지는가"가 코드 흐름에 보임 → 비개발자 추적 용이 | 호출 지점 누락 시 프로필 미생성 위험 → 단일 `ensureProfile` 헬퍼 + idempotent + 정해진 2지점 호출로 통제 |
| DB 트리거 `handle_new_user` (`auth.users` AFTER INSERT, SECURITY DEFINER) | 누락 원천 차단, 앱 코드 부담 0 | ① 신규 마이그레이션 `002` = DB 스키마 변경 → Hard Rule 8: 선행 ADR + 사용자 사전 승인 필요 ② `001`은 멱등이 아니라고 명시 — 트리거 추가/수정 실수 시 복구 난도↑ ③ 프로필 생성 로직이 DB 안에 숨어 코드만 읽어선 추적 불가(비개발자 사용자에 불리) ④ SECURITY DEFINER로 RLS를 우회 → `001`의 INSERT 정책이 무의미해짐(설계 불일치) |
| 브라우저(클라이언트)에서 직접 insert | 구현 단순 | 신뢰 경계 밖, 세션 직후 타이밍·재시도 문제. 서버에서 처리하는 것이 정석 |

결정적 근거 두 가지. **첫째**, `001_initial_schema.sql`은 `profiles`에 본인 한정 INSERT RLS 정책을 일부러 넣어 두었다 — 이는 "앱이 프로필을 만든다"는 전제로 설계됐다는 증거다. 트리거 방식은 이 정책을 무력화하므로 기존 설계와 어긋난다. **둘째**, 사용자(비개발자 풀스택 빌더)가 D-2에서 "코드 흐름 추적이 쉬워야 한다"고 명시했다. 트리거의 유일한 강점인 "누락 불가능"은 idempotent 헬퍼를 정해진 지점에서만 호출하는 규율로 충분히 대체된다.

### 3.3 세션 관리 — Supabase SSR 쿠키 세션

| 후보 | 장점 | 탈락 사유 |
|---|---|---|
| **`@supabase/ssr` 쿠키 세션 (선택)** | Next.js App Router의 서버 컴포넌트·Route Handler·미들웨어 어디서나 세션 접근. 이미 설치됨(`package.json`) + `lib/supabase/client.ts`·`server.ts` 구현 완료 | — |
| 클라이언트 전용 세션(`supabase-js` 기본, localStorage) | 설정 단순 | 서버 컴포넌트가 세션을 못 봄 → SSR 가드·미들웨어 보호 불가, 보호 페이지 깜빡임 |
| 자체 JWT·세션 직접 구현 | 완전한 제어 | 3개월 일정·비개발자 운영에 부적합, 보안 사고 표면↑ |

ADR-0001이 Next.js App Router + Supabase를 골랐으므로 공식 SSR 패턴이 정합이다. 토큰은 수명이 짧아 `middleware.ts`가 매 요청에서 갱신해 줘야 서버 컴포넌트가 만료된 세션을 만나지 않는다(결정 4의 미들웨어와 같은 지점에서 처리 → 일관성).

### 3.4 보호 라우트 — middleware 중앙 가드

| 후보 | 장점 | 탈락 사유 |
|---|---|---|
| **`middleware.ts` 중앙 가드 (선택)** | 보호/공개 판정이 한 파일. 보호 라우트 추가가 목록 한 줄. 캐시·렌더 이전에 실행돼 깜빡임 없음. 세션 갱신(결정 3)과 같은 곳에서 처리 | — |
| 페이지·레이아웃별 서버 가드 | 페이지마다 명시적 | phase-08~13 화면마다 `getUser()`+redirect 반복 → 누락 위험·중복 |
| 클라이언트 가드(`useEffect` 리다이렉트) | 구현 쉬움 | 보호 콘텐츠가 잠깐 노출됐다 사라짐, 비로그인에 화면 깜빡임 |

`tasks/_index.json`의 phase-07 검증 항목도 "`middleware.ts` 보호 라우트 작동"을 명시한다. **단, 미들웨어는 UX 가드일 뿐 데이터 보안의 최종 방어선이 아니다.** 미들웨어를 우회하더라도 남의 데이터는 RLS가 막는다(Hard Rule 6의 취지). 이 이중 구조 — 미들웨어(경험) + RLS(강제) — 를 설계 원칙으로 둔다.

## 4. 결과

- **DB 스키마 변경 없음** — `supabase/migrations/`에 새 파일이 생기지 않는다. Hard Rule 8의 사전 승인 절차가 발생하지 않는다. phase-07 검증 항목에 "마이그레이션 신규 파일 0건"을 포함한다.
- **새 환경변수 없음** — 기존 `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`만 사용한다. 단 `.env.local`에 publishable 키가 실제로 존재하는지 확인이 필요하다(phase-07 `user_steps`).
- **ADR-0001과의 관계** — 본 ADR은 ADR-0001 §2.2를 **대체(supersede)하지 않는다.** 베타 출시 스코프를 "이메일 + Google"로 좁히고 카카오를 연기하는 **구체화·일정 결정**이다. 카카오는 추후 ADR-0009 amendment 또는 후속 ADR로 다시 다룬다.
- **phase-08 연결** — "첫 로그인이면 `/onboarding`으로" 분기는 본 ADR의 보호 라우트·프로필 토대 위에 phase-08이 얹는다. phase-07의 임시 `/home`은 phase-10에서 교체된다.
- **미들웨어 + RLS 이중 방어**가 이후 모든 보호 화면(phase-08~13)의 전제가 된다.

---

## 5. 미반영 항목 (의도적 보류)

- **카카오 OAuth** — phase-13 이후 (결정 1).
- **`handle_new_user` DB 트리거** — 채택하지 않음 (결정 2). 프로필 누락 사고가 실제로 발생하면 §6 트리거로 재검토.
- **비밀번호 재설정·이메일 변경·계정 삭제 UI** — 베타 출시 후.
- **매직링크/OTP 로그인** — 미채택. 베타는 비밀번호 + Google로 한정.
- **다중 기기 세션 목록·강제 로그아웃 UI** — 범위 외.

---

## 6. 재검토 트리거

- 베타 피드백에서 카카오 로그인 요구가 분명한 이탈 요인으로 확인됨 → 공급자 추가.
- 앱-레벨 upsert 호출 지점 누락으로 `profiles` 행 누락 사고 발생 → DB 트리거 방식 재검토(그때는 Hard Rule 8에 따라 마이그레이션 ADR 선행).
- 비밀번호 분실 문의가 누적 → 비밀번호 재설정 흐름 별도 ADR.
- `@supabase/ssr` 메이저 변경 또는 Supabase Auth API 변경 → 세션 패턴 재검토.
