# ADR 0002 — 키키북스 디자인 시스템 채택

> **ADR (Architectural Decision Record)**: 디자인 시스템의 채택 근거와 대안 검토를 영구 기록한다.
> 미래에 "왜 이 시스템을 선택했는지", "왜 풀 핸드오프 번들이 아닌 시스템만 추출했는지" 답할 수 있는 단일 진실 공급원.

**문서 번호** ADR-0002
**상태** Accepted (확정)
**결정일** 2026-05-13
**관련 문서** `docs/design-system.md`, `docs/adr/0001-tech-stack.md`, `claude.md` 5절(라우팅)

---

## 1. 컨텍스트 (배경)

키키북스는 다음 제약 하에 디자인 시스템을 확정해야 한다.

- **사용자**: 한국 만 3~7세 자녀를 둔 학부모 (모바일 우선, 학습 결제 의사결정자)
- **개발자**: 비개발자 풀스택 빌더 1명 + Claude Code 자율 주행
- **빌드 환경**: Next.js 14 + Tailwind CSS + shadcn/ui (ADR-0001 확정)
- **타임라인**: 3개월 내 베타 출시, 14개 페이즈 자율 주행
- **법적 제약**: AttributionBox 등 라이선스 의무 컴포넌트가 디자인 시스템과 통합되어야 함 (`docs/guidelines/license-rules.md` 5절)

이 환경에서 다음 두 가지 핵심 질문에 답해야 했다.

1. **무엇을 채택할 것인가?** — 기존 디자인 시스템(Material 3, Apple HIG 등) vs Claude Design 추출본 vs 직접 설계
2. **어떻게 가져올 것인가?** — Claude Design 풀 핸드오프 번들 vs 시스템 토큰만 추출 vs Figma UI 키트

---

## 2. 결정 (선택한 방식)

### 2.1 무엇을: **Claude Design 추출본 + 키키북스 보강**

- 사용자가 Claude Design에서 작성한 시안에서 디자인 시스템 토큰만 추출
- 추출본 7개 영역(컬러·타이포·간격·라운드·그림자·컴포넌트 원칙·타겟 톤)을 그대로 채택
- 키키북스 특수 요구사항 5종을 보강:
  1. AttributionBox 컴포넌트 명세 (라이선스 의무 UI)
  2. Reader 모드 토큰 (content_type별 분기: HTML/ePub/H5P/PDF)
  3. Celebrate 화면 모션 토큰 (별 3개·포인트·배지 애니메이션)
  4. Streak 컴포넌트 토큰 (7일 막대그래프)
  5. 자녀 레벨 컬러 매핑 (Level 1~5)
- 다크 모드는 베타에서 미지원, 정책만 명시

### 2.2 어떻게: **시스템 토큰만 추출 (풀 핸드오프 번들 미사용)**

- Claude Design에서 화면 시안 전체를 export하는 핸드오프 번들은 사용하지 않는다
- 대신 시안에서 추상화된 디자인 시스템(`docs/design-system.md`)만 추출하여 영구 보관
- Claude Code는 화면 작업 시 시안이 아닌 **본 문서를 참조**하여 새롭게 구현한다

## 3. 대안 비교

### 3.1 무엇을 채택할 것인가

| 후보 | 장점 | 탈락 사유 |
|---|---|---|
| **Claude Design 추출본 + 보강 (선택)** | 사용자 의도 100% 반영, 한국 유아 학부모 톤 사전 검증, Tailwind 호환 토큰 | — |
| Material 3 Design System | Google 표준, 무료, 검증된 접근성 | 톤이 일반 SaaS에 가까워 유아 학부모 정서와 거리. 따뜻한 코랄·민트 톤 별도 커스터마이즈 부담 |
| Apple HIG (Human Interface Guidelines) | iOS 사용자에게 친숙 | 안드로이드 사용자(한국 70%+) 이질감. Tailwind와 직접 매핑 어려움 |
| Carbon (IBM), Polaris (Shopify) | 엔터프라이즈 검증 | B2B 톤, 유아 환경 부적합 |
| 직접 설계 (zero base) | 완전한 자유도 | 비개발자 + 3개월 일정에 비현실적. 토큰 시스템 설계만 2주 소요 추정 |

### 3.2 어떻게 가져올 것인가

| 후보 | 장점 | 탈락 사유 |
|---|---|---|
| **시스템 토큰만 추출 (선택)** | 컨텍스트 효율 1/5 수준, claude.md 지침 우선, 14개 페이즈 일관 적용 | — |
| Claude Design 풀 핸드오프 번들 | 시안 1:1 구현 빠름 | 핸드오프 번들 전체 로드 시 컨텍스트 소모 과다 (하네스 20% 원칙 위반). Claude Code가 시안에 끌려가 claude.md 우선 원칙 약화. 화면별 핸드오프 14회 반복 시 토큰 비용 폭증 |
| Figma UI 키트 → Claude Design → Code | 표준 디자인 시스템 활용 (Material 3 등) | 추가 단계로 일정 지연. Figma 라이선스 비용 발생. 비개발자에게 학습 곡선 |
| 시안 캡처 → 수동 토큰 작성 | 가장 단순 | 토큰 정밀도 낮음. 색상 정확도 손실. 추후 보정 비용 |

## 4. 결정 근거

### 4.1 하네스 가이드라인 정렬

본 결정은 하네스 엔지니어링 가이드라인의 3가지 핵심 원칙과 정렬된다.

| 원칙 | 본 결정의 반영 |
|---|---|
| **컨텍스트 20% 미만 유지** | 핸드오프 번들(~30% 컨텍스트) 대신 추출본(~5%) 사용 |
| **암묵지의 명시적 변환** | 사용자의 비주얼 안목을 토큰 7개 영역 + 보강 5종으로 명시화 |
| **문서 우선 원칙** | 화면 작업 전 design-system.md를 먼저 참조하도록 라우팅 |

### 4.2 비개발자 사용자에게 적합

- 사용자가 직접 Tailwind 토큰을 손볼 필요 없음 (Claude Code가 자동 적용)
- 디자인 변경 필요 시 `design-system.md` 한 파일만 수정하면 14개 페이즈 전체 반영
- 추출본의 한국어 설명이 사용자의 의도 검증을 용이하게 함

### 4.3 키키북스 특수 요구 충족

기존 디자인 시스템(Material 3 등)에는 없는 4가지 특수 컴포넌트가 보강되었다.

| 보강 항목 | 필요 사유 |
|---|---|
| AttributionBox | CC BY 4.0 법적 의무 표시 (`license-rules.md` 5절 연동) |
| Reader 모드 토큰 | content_type 분기 처리 (HTML/ePub/H5P/PDF) |
| Celebrate 모션 토큰 | 유아 완독 보상 UX의 핵심 모먼트 |
| 자녀 레벨 컬러 매핑 | 5단계 학습 진행을 시각화 (난이도 ≠ 부정 신호) |

## 5. 결과 및 영향 (Consequences)

### 5.1 긍정적 결과

- **컨텍스트 효율**: 화면 작업당 토큰 사용량 60~70% 절감 예상
- **일관성**: 14개 페이즈 어디서든 동일 토큰 참조 → 디자인 드리프트 방지
- **법적 안전성**: AttributionBox가 디자인 시스템에 통합되어 누락 가능성 감소
- **유지보수**: 시스템 변경 시 단일 파일 수정으로 전파
- **Tailwind 통합**: 10절의 매핑 가이드로 Claude Code가 자동 변환 가능

### 5.2 부정적 결과 / 트레이드오프

| 트레이드오프 | 영향 | 완화 방안 |
|---|---|---|
| 시안 그대로 복제 못함 | 사용자가 본 시안과 실제 화면이 다를 수 있음 | 화면 작업 후 design-system.md 미세 보정으로 시안 톤에 근접 |
| 다크 모드 미지원 | 다크 선호 사용자 일부 이탈 가능 | Phase 2 검토 트리거 정의 (피드백 10건+) |
| 일부 토큰 범위 → 단일값 고정 | Tailwind 호환을 위한 정밀도 손실 | 예외 필요 시 ADR 추가 작성 |
| 한글 폰트 의존 (Pretendard) | CDN 다운 시 폴백 필요 | system-ui 폴백 체인 명시 |

### 5.3 향후 검토 트리거

다음 조건 발생 시 본 ADR을 재검토하고 신규 ADR을 작성한다.

| 트리거 | 검토 대상 |
|---|---|
| 베타 사용자 다크 모드 요청 10건+ | 다크 모드 토큰 추가 ADR |
| MAU 1,000명 + 디자이너 합류 | Figma 디자인 시스템으로 마이그레이션 검토 |
| 새로운 디바이스 지원 (TV, 워치) | 디바이스별 토큰 분기 ADR |
| 다국어 확장 (영어·일본어) | 폰트 페어링 ADR |
| B2B 학원 대시보드 별도 톤 필요 | 멀티 브랜드 시스템 ADR |
| Claude Design 시안 대규모 변경 | design-system.md v2.0 마이그레이션 ADR |

## 7. 미래 자신을 위한 메모

비개발자 사용자가 이 ADR을 읽을 때 알아두면 좋은 것:

1. **이 ADR은 "왜 시안이 아니라 시스템을 가져왔는지"의 영구 답변**이다. 나중에 "그냥 시안 그대로 쓰지 그랬어?"라는 후회가 들면 본 문서 3.2절을 읽어라.
2. **시안과 실제 화면이 다르게 보일 수 있다.** 그것은 버그가 아니라 의도된 트레이드오프다 (5.2절).
3. **디자인 시스템 변경은 신중하게.** 한 토큰 수정이 14개 페이즈 전체에 전파된다. 색상 한 줄 바꾸기 전에 임팩트를 생각하라.
4. **다크 모드는 베타에서 안 한다.** 사용자가 요청해도 Phase 2까지 보류한다 (5.3 트리거).
5. **AttributionBox는 디자인 시스템의 일부이자 법적 의무다.** 디자인 사유로 숨기거나 축소하지 않는다.
