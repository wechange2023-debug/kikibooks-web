## 1. 이 문서의 목적

이 문서는 키키북스의 **로그인·회원가입·세션·보호 라우트**가 사용자 입장에서 어떻게 동작해야 하는지를 자연어로 먼저 못박는다. 코드는 이 문서를 따른다(claude.md §3-5: 의도 문서 선행). 기술적 "왜"의 근거와 대안 비교는 `docs/adr/0009-auth-architecture.md`에 있다.

## 2. 범위

**phase-07이 다루는 것**

- 이메일(비밀번호) 회원가입·로그인
- Google 소셜 로그인
- 세션 유지(로그인 상태 기억)와 자동 갱신
- 보호 라우트 가드(비로그인 사용자 차단)
- 로그아웃
- 로그인한 사용자의 `profiles` 행 자동 생성

**phase-07이 다루지 않는 것 (다음 페이즈로 연결)**

- 카카오 로그인 → phase-13 이후 (ADR-0009 결정 1)
- 자녀 프로필 등록 온보딩(`/onboarding`) → phase-08
- "첫 로그인이면 온보딩으로" 분기 → phase-08
- 5개 화면(랜딩·홈·책 상세·뷰어·완독 보상) → phase-09~13
- 비밀번호 재설정·이메일 변경 등 계정 관리 → 베타 출시 후

## 3. 라우트 지도

| 경로 | 공개/보호 | 비고 |
|---|---|---|
| `/` | 공개 | 랜딩 (phase-09에서 구현, 지금은 phase-01 임시 화면) |
| `/login` | 공개 | 로그인 페이지 — 이미 로그인 상태면 `/home`으로 보냄 |
| `/signup` | 공개 | 회원가입 페이지 — 이미 로그인 상태면 `/home`으로 보냄 |
| `/auth/callback` | 공개 | 이메일 확인 링크·Google 로그인이 되돌아오는 진입점 |
| `/auth/auth-error` | 공개 | 콜백 실패 안내 |
| `/home` | 보호 | phase-07에서는 "로그인됨" 임시 화면. phase-10에서 진짜 홈으로 교체 |
| `/onboarding` | 보호 | phase-08에서 구현 |
| `/library`, `/book/...` | 보호 | phase-11~13에서 구현 |

"보호"는 로그인하지 않으면 들어갈 수 없다는 뜻이다. 차단은 `middleware.ts`가 담당한다(§4.5).

## 4. 사용자 흐름 (단계별)

### 4.1 이메일 회원가입 → 프로필 생성 → 보호 라우트 진입 (핵심 흐름)

1. 비로그인 학부모가 `/signup`에 들어온다. `/signup`은 공개 라우트라 미들웨어가 통과시킨다.
2. 이메일과 비밀번호를 입력하고 "회원가입"을 누른다. 화면의 폼은 입력값을 **서버 액션**으로 보낸다(브라우저가 비밀번호를 직접 들고 다니지 않는다).
3. 서버 액션이 Supabase에 회원가입을 요청한다. Supabase가 사용자에게 **확인 메일**을 보낸다. 화면에는 "메일함을 확인해 인증 링크를 눌러주세요"라고 안내한다.
4. 사용자가 메일 속 링크를 누르면 브라우저가 `/auth/callback?code=...`로 이동한다.
5. `/auth/callback`이 받은 `code`를 **세션으로 교환**한다. 이 순간부터 브라우저에 로그인 상태를 나타내는 **세션 쿠키**가 생긴다.
6. 세션이 생기자마자 콜백이 **프로필 생성 절차**(§4.4)를 실행한다 — `profiles` 테이블에 이 사용자의 행이 없으면 새로 만든다.
7. 콜백이 사용자를 `/home`으로 보낸다. (phase-08부터는 "자녀 프로필이 없으면 `/onboarding`으로" 분기가 이 지점에 추가된다.)
8. 이후 사용자가 `/home` 같은 보호 라우트를 열 때마다 미들웨어가 세션 쿠키를 확인한다. 유효하면 통과, 없거나 만료면 `/login`으로 돌려보낸다.

> 참고: Supabase에서 이메일 확인 옵션을 끄면 3~5단계가 생략되고 회원가입 직후 바로 세션이 만들어진다. 이 경우 프로필 생성(§4.4)은 회원가입 서버 액션이 직접 실행한다. **두 경로 모두 프로필 생성 절차를 반드시 거친다.**

### 4.2 이메일 로그인

1. 이미 가입한 사용자가 `/login`에서 이메일·비밀번호를 입력하고 "로그인"을 누른다.
2. 서버 액션이 Supabase에 로그인을 요청한다. 성공하면 세션 쿠키가 발급된다.
3. 프로필 생성 절차(§4.4)를 한 번 더 확인 호출한다. 이미 행이 있으면 아무 일도 일어나지 않는다(안전).
4. `/home`으로 이동한다.
5. 비밀번호가 틀리면 폼 위에 "이메일 또는 비밀번호가 올바르지 않습니다"라고 안내한다. 어느 쪽이 틀렸는지는 알려주지 않는다(계정 존재 여부 노출 방지 — 보안).

### 4.3 Google 로그인

1. `/login` 또는 `/signup`에서 "Google로 계속하기" 버튼을 누른다.
2. 브라우저가 Google 동의 화면으로 이동한다. 사용자가 계정을 선택하고 동의한다.
3. Google이 사용자를 `/auth/callback?code=...`로 되돌려보낸다.
4. 이후는 §4.1의 5~8단계와 동일하다(세션 교환 → 프로필 생성 → `/home`).
5. Google 로그인은 별도 회원가입 단계가 없다 — 처음 누른 사람은 그 자리에서 계정이 생기고, 프로필도 §4.4로 함께 만들어진다.

### 4.4 프로필 생성 — 앱-레벨 upsert (ADR-0009 결정 2)

- `profiles` 테이블의 행은 Supabase 인증 사용자(`auth.users`)와 1:1로 짝지어지고, 같은 `id`를 공유한다(`001_initial_schema.sql`).
- 키키북스는 이 행을 **DB 트리거가 아니라 앱 코드가** 만든다. 이유와 대안 비교는 ADR-0009 §3.2.
- 동작: 로그인으로 세션이 처음 생기는 순간, `ensureProfile`이라는 **단일 헬퍼**가 `profiles`에 `{ id = 사용자 id, email = 사용자 이메일 }`을 **있으면 그대로 두고 없으면 추가**하는 방식(idempotent upsert)으로 한 줄을 보장한다.
- 이 헬퍼는 **사용자 본인 세션**으로 실행되므로 `001` 스키마에 이미 들어 있는 RLS 정책 `users can insert own profile`을 그대로 통과한다 — 추가 권한이나 스키마 변경이 필요 없다.
- 여러 번 호출돼도 안전하다(이미 행이 있으면 아무 일도 안 함). 그래서 §4.1·§4.2·§4.3 어디서 호출되든 문제가 없다.
- `display_name`은 비운 채 시작한다. 이후 phase-08 온보딩이나 계정 설정에서 채운다. `role`은 DB 기본값 `parent`.
- 호출 지점은 **최소화**한다: ① `/auth/callback`(이메일 확인 링크·Google), ② 이메일 회원가입/로그인 서버 액션. 정해진 지점에서만 한 헬퍼를 부른다.

### 4.5 보호 라우트 진입과 차단

- `middleware.ts`가 **모든 요청**에서 두 가지를 한다.
  1. 세션 쿠키를 갱신한다(토큰이 곧 만료될 경우 새로 받아, 서버 컴포넌트가 만료된 세션을 보지 않게 한다).
  2. 요청 경로를 판정한다.
     - 보호 라우트(`/home`, `/onboarding`, `/library`, `/book/...`)인데 세션이 없으면 → `/login`으로 보낸다.
     - `/login`·`/signup`인데 이미 세션이 있으면 → `/home`으로 보낸다(로그인한 사람에게 로그인 폼을 다시 보여주지 않는다).
     - 그 외 공개 경로(`/`, `/auth/...`)는 통과시킨다.
- **중요**: 미들웨어는 *화면 차원의* 가드다. 데이터 자체의 보호는 DB의 **RLS**가 책임진다. 누가 미들웨어를 우회하더라도 남의 자녀·세션 데이터는 RLS가 막는다(claude.md Hard Rule 6의 취지). 미들웨어는 사용자 경험(엉뚱한 화면 깜빡임 방지)을 위한 것이고, **RLS가 최종 방어선**이다.

### 4.6 로그아웃

1. 로그인 상태에서 "로그아웃"을 누른다.
2. 서버가 세션을 종료하고 세션 쿠키를 지운다.
3. 사용자를 `/`(랜딩)로 보낸다.
4. 이후 보호 라우트를 다시 열면 §4.5에 따라 `/login`으로 차단된다.

## 5. 지켜야 할 규칙·제약

- **Hard Rule 6** — Supabase 비밀 키(`SUPABASE_SECRET_KEY`)는 인증 흐름 어디에도 쓰지 않는다. 로그인·세션은 전부 publishable 키(`NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`)로만 동작한다. 비밀 키는 클라이언트 코드·공개 환경변수에 등장 금지.
- **Hard Rule 10** — `/login`·`/signup` 화면의 색·간격·폰트는 `docs/design-system.md`의 semantic 토큰만 사용한다. raw 색상값 직접 입력 금지.
- **비밀번호는 서버 액션으로** — 비밀번호 입력 처리는 서버 액션이 담당한다.
- **프로필 생성은 idempotent** — §4.4. 누락도 중복도 사고가 되지 않게 한다.
- **오류 메시지는 계정 존재 여부를 노출하지 않는다** — §4.2.
