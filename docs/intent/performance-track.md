## 1. 이 문서의 목적

팀장이 "플랫폼이 조금씩 느려지는 체감"을 보고했다. 읽기전용 진단
(`docs/review/2026-07-01-senior-code-review-performance.md`)이 원인을 **"요청당 왕복(round-trip)
수 + 캐싱 부재의 곱"**으로 짚었다(죽은 코드가 아니다 — Next tree-shaking).

본 문서는 그 진단을 실제 개선으로 옮기는 **단계적 트랙**의 의도를 자연어로 못박는다.
코드는 이 문서와 진단 리포트를 따른다(claude.md §3-5: 의도 문서 선행).

트랙의 목표는 하나다: **화면 출력은 100% 불변으로 유지하면서, 요청당 DB/Auth 왕복 수와
불필요한 반복 계산을 단계적으로 줄인다.**

---

## 2. 착수 순서 (전체 트랙)

진단 리포트 P0 4건을 위험도·선행조건 순으로 배열한다. **한 번에 하나씩, 별도 라운드로**
진행한다(회귀 표면 최소화).

| 순서 | 항목 | 성격 | 선행조건 |
|---|---|---|---|
| 1 | **P0-3** 홈 순차 왕복 병렬화 + `getCategoryDistribution` 반복 계산 제거 | 구조 개선(캐싱 무관) | 없음 — **이번 라운드** |
| 2 | P0-4 라이브러리 count 첫 페이지만 계산 | 국소 수정 | 없음 |
| 3 | P0-2 auth 중복 검증(요청당 getUser 2회) 완화 | 보안 경계 | ADR 선행 + RLS 재확인 |
| 4 | P0-1 force-dynamic 제거 + 카탈로그 캐싱 도입 | 아키텍처 | **ADR 선행 필수** |

원칙: 캐싱(P0-1)은 가장 효과가 크지만 자녀-무관 데이터 한정 설계가 필요해 **맨 마지막**,
ADR을 먼저 쓴다. auth(P0-2)는 보안 경계라 신중히. 구조 개선(P0-3)과 국소 수정(P0-4)은
동작 불변이 명확해 먼저 착수한다.

## 3. 이번 라운드 범위 (P0-3, 캐싱 미포함 부분만)

**포함**
- **(A) 홈 쿼리 병렬화** — `app/(reader)/home/page.tsx`의 데이터 페칭 중, 선행 결과에
  의존하지 않는 순차 쿼리를 `Promise.all`로 병렬화한다. 선행 의존이 있는 쿼리는
  병렬화하지 않는다(정합성 우선).
- **(B) `getCategoryDistribution` 반복 계산 제거** — 매 홈 로드마다 활성 전권 × 108키워드
  ≈ 95,000회 제목 substring 매칭을 실행하는 구조를 개선한다. 단일 SQL 집계로 1:1 대체가
  가능하면 그렇게 하고, 제목 기반이라 어려우면 **최소한 요청당 1회만 계산**(중복 제거)한다.

**절대 제외 (이번 라운드 금지)**
- `force-dynamic` 설정 변경 / 제거
- 캐싱·`revalidate`·`unstable_cache` 도입 → P0-1 트랙(ADR 선행)
- `auth.getUser()` 관련 변경 → P0-2 트랙

**불변식 (게이트)**
- 화면 출력은 기존과 100% 동일. 성능만 바꾸고 결과는 불변.
- (B)는 **개선 전/후 "카테고리별 count"가 완전히 동일함을 대조 검증한 뒤에만** 반영한다.
  동일하지 않으면 반영하지 않고 차이를 보고한다.
